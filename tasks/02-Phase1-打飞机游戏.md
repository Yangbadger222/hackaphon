# 任务 02：Phase 1 — 打飞机游戏

## 开发目的
实现一个极简的像素风纵向射击游戏作为"伪装"。玩家击杀第 10 个敌机时触发觉醒动画，然后切换到 Phase 2。游戏必须看起来非常廉价、普通——这样后续 AI 觉醒的反转才有冲击力。

## 前置依赖
- 任务 01 完成（项目骨架、GameContext、App.jsx 路由就绪）

## 需要完成的事项

### 1. ShooterGame 组件 — `src/components/ShooterGame.jsx`

**画面布局：**
- 全屏黑色背景，居中显示一个 800×600 的 Canvas
- Canvas 上方放一个假窗口标题栏，文字 `H.O.R.S.E._v0.1.exe`，右侧 `─ □ ×`，灰色背景灰色文字，font-mono，制造桌面应用伪装感

**游戏对象（全部用最简单的几何图形）：**
- 玩家飞船：白色三角形，约 30×30px，位于底部
- 敌机：白色方块，约 20×20px，从顶部随机 X 坐标生成，匀速下落
- 子弹：白色矩形，4×12px，向上移动
- 爆炸粒子：8 个白色 2×2px 小点向四周扩散，20 帧生命周期

**操作方式：**
- 左右方向键 / A D 键移动飞船
- 空格键发射子弹（限制射速：每 200ms 最多一发）
- 飞船不能移出画布边界

**游戏循环（帧率无关）：**
- 用 `requestAnimationFrame` 驱动，每帧计算 `deltaTime = (now - lastFrameTime) / 1000`（秒），用 `performance.now()` 取时间戳
- 所有运动量乘以 deltaTime：`position += speed * PIXELS_PER_SECOND * deltaTime`
- 基准速度常量用"像素/秒"定义，不要用"像素/帧"：
  - 玩家移动：`PLAYER_SPEED = 360` px/s
  - 敌机下落：`ENEMY_SPEED_BASE = 120` px/s（每个敌机 +随机 0~60）
  - 子弹上升：`BULLET_SPEED = 480` px/s
- 粒子生命周期也用秒：`life` 初始 0.4s，每帧减 `deltaTime`，≤0 时移除
- 防止 deltaTime 极端值（切标签页回来）：`deltaTime = Math.min(deltaTime, 0.1)`
- 敌机生成用累加器：`spawnTimer += deltaTime`，达到 1.5s 时生成一个并重置
- 子弹和敌机的碰撞用矩形相交检测
- 命中后：敌机移除、子弹移除、生成爆炸粒子、score++、调用 `dispatch({ type: 'INCREMENT_SCORE' })`
- 左上角显示 `SCORE: X`，灰色 14px monospace

### 2. 埋点异常系统

在击杀回调中根据 score 触发微妙异常。

**关键原则：** 异常必须与"性能卡顿"在视觉上有明确区分。真正的卡顿是画面完全静止、输入无响应；而游戏中的异常应伴随**主动的视觉抖动/故障特征**，让玩家感知到"有什么东西不对"而非"游戏卡了"。

| 击杀数 | 异常 | 如何区别于卡顿 |
|--------|------|---------------|
| 6 | 在 Canvas 右下角绘制红色 `"...neigh..."` 字符，100ms 后被下一帧覆盖 | 有明确的红色视觉元素出现 |
| 7 | 所有敌机"颤抖定格"0.5 秒：速度归零但每帧给每个敌机添加 ±1~2px 的随机抖动偏移（仅影响绘制坐标，不改变逻辑坐标），同时玩家飞船和子弹**继续正常运行** | 卡顿 = 全部冻结；异常 = 只有敌机抖动，其余正常 |
| 8 | 标记一个 flag，下次碰撞时爆炸粒子颜色改为红色（仅一次） | 有明确的颜色变化 |
| 9 | 被击中的敌机延迟 0.5 秒消失：先原地抖动（每帧 ±2px 随机偏移）+ 闪烁（每隔 3 帧切换透明/不透明），然后消失 | 卡顿 = 静止不动；异常 = 抖动+闪烁的主动特效 |
| 10 | 触发觉醒序列 | — |

**抖动实现方式：** 在绘制阶段，对需要抖动的对象的渲染坐标叠加 `Math.random() * jitterRange - jitterRange/2`，逻辑坐标不变。这样碰撞检测不受影响，只有视觉表现异常。

### 3. 觉醒触发序列

击杀第 10 个敌机后，按以下时间线执行。

**核心要求：向心缩放的对象是包含标题栏在内的整个游戏画面的快照，不是只缩 Canvas。**

实现方式：用 `html2canvas` 或手动方案对包裹标题栏和 Canvas 的整个容器做快照，然后对该快照执行向心缩放动画。

**手动快照方案（不依赖第三方库）：**
- 标题栏和 Canvas 共同包裹在一个容器 div 中（ref 为 `containerRef`）
- 觉醒触发时，在容器上方叠加一个与容器等大的 `<canvas id="snapshotCanvas">`
- 将原 Canvas 内容用 `drawImage(gameCanvas, ...)` 绘制到快照 Canvas 对应位置
- 标题栏区域用 `fillRect` + `fillText` 模拟绘制（灰色背景、文字 `H.O.R.S.E._v0.1.exe` 和 `─ □ ×`）
- 快照 Canvas 尺寸 = 容器尺寸（包含标题栏高度 + 游戏画布高度）
- 然后隐藏原容器，只显示快照 Canvas，对快照执行向心缩放

```
T+0.0s   停止 requestAnimationFrame 游戏循环
T+0.1s   所有敌机进入抖动定格状态（同击杀 7 的抖动效果），玩家飞船和子弹也冻结
T+0.3s   红色半透明覆盖闪烁 3 次（每次 100ms 亮 100ms 灭），闪烁覆盖整个容器（包含标题栏）
T+0.9s   生成全画面快照（含标题栏），隐藏原容器，显示快照 Canvas
T+1.0s   向心缩放动画开始：用 requestAnimationFrame 驱动，每帧执行：
         1. 清空快照 Canvas 为纯黑
         2. ctx.save()
         3. ctx.translate(centerX, centerY)  // 中心点 = 快照 Canvas 宽高各一半
         4. ctx.scale(currentScale, currentScale)
         5. ctx.translate(-centerX, -centerY)
         6. ctx.drawImage(snapshotImage, 0, 0)  // snapshotImage 是冻结时刻的快照
         7. ctx.restore()
         8. currentScale -= 0.03
         9. 当 currentScale <= 0 时停止，填充纯黑
T+2.0s   缩放完成，全屏纯黑
T+2.5s   调用 dispatch({ type: 'SET_PHASE', payload: PHASES.GLITCH }) 切换到 Phase 2
```

**向心缩放的视觉效果：** 整个"窗口"（标题栏 + 游戏画面）像被黑洞吞噬一样向中心点缩小并消失，背景始终是纯黑。这比只缩 Canvas 更有"这个应用正在被摧毁"的感觉。

## 约束
- 只用 Canvas 2D API 绘制，不要引入任何游戏框架或精灵图
- 所有图形用 `fillRect`、`beginPath`/`fill` 等基础 API
- 颜色只用黑白（背景黑、所有元素白），除了埋点异常用红色
- 不要做积分系统、生命值、关卡等额外功能
- 不要向后端发请求——Phase 1 是纯前端的
- 不要做任何移动端适配，桌面端优先

## 验证方法
1. 打开页面，看到黑色画布和假窗口标题栏
2. 按方向键飞船左右移动，不会超出画布
3. 按空格发射子弹，子弹向上飞
4. 敌机从顶部不断生成并向下掉落
5. 子弹击中敌机后两者消失，出现白色粒子效果，左上角分数+1
6. **帧率无关验证：** 在 60Hz 和 144Hz 显示器上（或用浏览器限帧工具），敌机下落速度和子弹速度目测一致，游戏体验没有明显差异
7. 击杀第 6 个时右下角短暂闪过红色 `?`
8. 击杀第 7 个时所有敌机**抖动定格** 0.5 秒（敌机在原位小幅抖动），但飞船和子弹仍可正常操作——这与浏览器卡顿（全部静止、无法操作）有明显区别
9. 击杀第 9 个时，被击中的敌机原地抖动+闪烁 0.5 秒后消失
10. 击杀第 10 个时：全画面（包含标题栏）冻结抖动 → 红闪覆盖整个窗口（含标题栏）→ 整个窗口画面作为一个整体向中心缩小消失 → 纯黑 → 页面切换到 Phase 2
11. 向心缩放过程中标题栏文字 `H.O.R.S.E._v0.1.exe` 随画面一同缩小，不会残留在原位
12. 整个过程 console 无报错
